## Defects found and repairs

### Migrations 0042–0044

All three SQL files applied successfully in order on an empty local test
database. The defect was schema drift in their foreign-key constraint names:
inline PostgreSQL-generated names did not match the names generated by the
current Drizzle model. Because the production migration ledger explicitly
confirmed that 0042–0044 had not been applied to any shared environment, those
three files were repaired in place with explicit Drizzle-compatible constraint
names. Their table, column, index, check and foreign-key behavior was not
otherwise changed.

Each file now documents that all of its statements are transaction-safe. The
automated verifier executes every numbered SQL migration inside an explicit
`BEGIN`/`COMMIT`, rolls back on failure, and rejects known non-transactional
statements. No migration required a non-transactional exception.

### Full-chain drift found outside 0042–0044

The complete migration-to-model comparison also found two historical runtime
mismatches that could not safely be repaired by editing migrations that may
already have been applied:

1. `customer_timeline_events.projected_at` existed in migrated databases while
   the runtime model expects `created_at`. Migration 0045 performs a guarded,
   data-preserving rename and stops for manual reconciliation if both columns
   unexpectedly exist.
2. `tenants.invoice_bank_currency` was migrated as constrained `text` while the
   runtime model uses the existing `currency` enum. Migration 0045 removes the
   superseded check and performs an explicit cast to the enum, stopping on an
   unexpected source type.

The Drizzle model was also aligned to already-shipped database behavior for
nullable idempotency partial indexes, the ledger-line `ON DELETE RESTRICT`
foreign key, and subscription-payment-attempt RLS. These were model-definition
mismatches, not new database behavior.

## In-place versus forward-migration decision

- **In place:** 0042, 0043 and 0044 only, because the authoritative production
  ledger states that none had been applied to a shared environment.
- **Forward migration:** new migration **0045** for every correction involving
  historical objects that may already exist in shared environments. It is
  additive/data-preserving and contains no drop of a data-bearing object.
- **Next free migration number:** **0046**.

## Automated migration and drift control

The quality workflow now checks out full history, reconstructs the historical
baseline, applies every contiguous numbered migration from 0000 through 0045
to a fresh local PostgreSQL database, and creates a separate reference database
from the current Drizzle model. It compares tables and RLS, columns and types,
defaults, constraints and foreign keys, indexes, and enums. Any migration
failure, non-transactional statement, numbering gap, or structural drift fails
CI.

The deployment-time schema check remains strict for enabled features while
preserving the build-dark release contract: 0042–0044 columns are required only
when `migration.hub`, `network.directory`, or `documents.workspace` is enabled
in the target database.
